# 任务工作流闭环 V1 设计

## 背景

SD Kanban 已经具备登录、项目、项目负责人、项目成员、默认看板列、项目总体看板、个人任务看板、任务创建、拖拽移动、任务详情与评论能力。当前缺口是任务创建后的日常维护链路还不完整：用户不能在前端直接编辑任务主要字段，也不能从界面完成、归档或删除任务。这会让看板停留在“能展示”而不是“能持续使用”的状态。

本设计聚焦任务工作流闭环 V1，不扩展标签管理、迭代管理、批量导入导出或复杂权限模型。

## 目标

让项目成员能在项目看板中完成一条任务的核心生命周期：

1. 创建任务。
2. 打开任务详情。
3. 编辑任务信息。
4. 分配或取消负责人。
5. 调整优先级、类型、故事点、预计工时、截止日期和验收标准。
6. 将任务标记完成。
7. 归档或删除任务，使其从日常看板中消失。
8. 通过评论和动态记录追踪关键变更。

## 非目标

- 不实现标签配置和标签选择增强。
- 不实现迭代创建、迭代选择增强和冲刺燃尽图。
- 不实现批量移动、批量删除、导入导出。
- 不新增复杂角色权限；继续沿用现有“项目成员可操作项目任务”的规则。
- 不新增真实物理删除；删除仍为软删除。

## 用户体验

### 项目看板

项目看板继续作为任务工作的主入口。任务卡片在现有标题、类型、优先级、故事点和截止日期基础上，增强以下信息：

- 负责人昵称；未分配时显示“未分配”。
- 截止日期状态；逾期且未完成时给出明显但克制的提示。
- 空列继续显示“暂无任务”，并保留列级“+任务”入口。

看板筛选增加负责人筛选。负责人选项来自项目成员列表，支持“全部负责人”和“未分配”。

### 任务详情抽屉

任务详情抽屉升级为任务工作台，包含两个模式：

- 查看模式：展示任务信息、描述、验收标准、标签、评论和动态。
- 编辑模式：在同一抽屉内编辑任务字段，保存后回到查看模式。

编辑字段：

- 标题，必填，最多 200 字符。
- 描述，可清空。
- 任务类型：`TASK`、`STORY`、`BUG`。
- 优先级：`HIGH`、`MEDIUM`、`LOW`。
- 负责人：项目成员或未分配。
- 故事点，可清空。
- 预计工时，可清空。
- 截止日期，可清空。
- 验收标准，可清空。

操作按钮：

- 保存：提交修改，成功后刷新详情和看板卡片。
- 标记完成：将任务移动到项目第一个 `isDone=true` 的看板列。
- 归档：软归档任务，并从项目看板和我的任务看板中隐藏。
- 删除：二次确认后软删除任务，并从看板中隐藏。

如果项目没有完成列，隐藏或禁用“标记完成”，并给出“项目暂无完成列”的提示。

## 后端设计

### 复用现有能力

继续复用：

- `PATCH /api/tasks/{taskId}` 更新任务字段。
- `PATCH /api/tasks/{taskId}/position` 移动任务列和排序。
- `GET /api/projects/{projectId}/board` 获取项目看板。
- `GET /api/tasks/mine/board` 获取个人任务看板。

### 新增任务归档和删除能力

新增接口：

- `PATCH /api/tasks/{taskId}/archive`
  - 将 `is_archived` 设为 `true`。
  - 记录 `TASK_ARCHIVED` 动态。
  - 返回更新后的任务响应；前端随后从当前看板中移除该任务。

- `DELETE /api/tasks/{taskId}`
  - 将 `is_deleted` 设为 `true`。
  - 记录 `TASK_DELETED` 动态。
  - 返回空响应。

归档和删除都必须校验当前用户是任务所属项目成员。

### 清空字段语义

现有 `UpdateTaskRequest` 使用 `null` 表示字段未参与更新，因此无法区分“未传字段”和“显式清空字段”。V1 采用兼容方案：为更新请求增加 `clearFields: string[]`。

允许的清空字段：

- `description`
- `storyPoints`
- `estimatedHours`
- `dueDate`
- `acceptanceCriteria`
- `assigneeId`
- `sprintId`

服务端处理顺序：

1. 先处理 `clearFields`，把指定字段写为 `null` 并记录动态。
2. 再处理请求中非 `null` 的字段更新。
3. 如果同一个字段既在 `clearFields` 又传了非空值，非空值优先生效，避免前端表单状态竞态导致误清空。

### 看板过滤

项目看板、个人任务看板、仪表盘统计必须排除 `is_deleted=true` 和 `is_archived=true` 的任务。本轮实现时需要逐一核对这些查询，并为缺少归档过滤的查询补齐条件。

## 前端设计

### API 层

`web/src/api/tasks.ts` 增加：

- `archiveTask(taskId)`
- `deleteTask(taskId)`
- `UpdateTaskRequest.clearFields`

`web/src/api/board.ts` 的 `TaskCard` 增加足够展示卡片增强的信息：

- `assignee`，使用与任务详情一致的 `UserSummary | null` 结构；保留 `assigneeId` 仅用于兼容现有逻辑。
- `isOverdue` 不由后端新增字段计算，前端基于 `dueDate` 和列 `isDone` 判断。

### Store 层

`useTasksStore` 增加：

- 编辑草稿保存动作，调用 `updateTask`。
- 归档动作，调用 `archiveTask`。
- 删除动作，调用 `deleteTask`。
- 操作成功后更新 `activeTask` 或关闭抽屉。

`useBoardStore` 增加：

- 当前项目看板刷新辅助，保留现有筛选条件。
- 从当前看板移除任务的动作，用于归档/删除后立即反馈。
- 标记完成时复用移动任务逻辑，目标列为第一个完成列。

### 组件层

`TaskDrawer.vue` 拆出或内聚以下职责：

- 查看状态展示任务详情。
- 编辑状态维护本地草稿。
- 保存时显示提交中状态。
- 保存失败时保留草稿并显示错误。
- 归档和删除失败时显示错误。

`ProjectBoardView.vue` 负责把项目成员、看板列、当前筛选条件传给抽屉和创建弹窗。任务变更后刷新看板或局部移除卡片。

## 错误处理

- 保存失败：留在编辑模式，保留用户输入，显示“任务保存失败，请重试”。
- 标记完成失败：任务留在原列，显示“任务状态更新失败，请重试”。
- 归档失败：任务仍在看板，显示“任务归档失败，请重试”。
- 删除失败：任务仍在看板，显示“任务删除失败，请重试”。
- 项目成员加载失败：负责人下拉退化为只读展示，编辑时仍可不修改负责人。

- `TaskControllerTest` 增加删除任务测试：项目成员删除后，详情接口返回 not found，看板不再返回该任务，数据库 `is_deleted=true`。

## 验收标准

- 用户可以在看板中创建任务后继续编辑该任务。
- 用户可以把任务分配给项目成员，也可以取消分配。
- 用户可以清空可选字段，包括负责人、截止日期、故事点和预计工时。
- 用户可以一键将任务标记完成，任务移动到 Done 类列。
- 用户可以归档任务，任务从项目看板和个人任务看板隐藏。
- 用户可以删除任务，删除前必须确认，删除后任务从看板隐藏且详情不可再打开。
- 所有新增行为都有自动化测试覆盖，并通过前端测试、后端相关测试、构建和浏览器实测。
